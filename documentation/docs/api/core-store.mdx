# API: `createStore`

The `createStore` function is the foundation of SoulState. It creates a store instance that holds your application state and provides methods to read and update it.

## `createStore`

Creates a new store instance. It can take either an initial state object or a "creator" function.

### Signature

<CodeBlock language="typescript">
{`function createStore<T extends State>(
  creator: StateCreator<T> | T
): StoreApi<T>;
`}
</CodeBlock>

- **`creator`**: The initial state object or a function to create the state.
- **Returns**: A `StoreApi<T>` instance.

### Using an Initial State Object

<CodeBlock language="typescript">
{`import { createStore } from 'soulstate';

export const counterStore = createStore({
  count: 0
});
`}
</CodeBlock>

### Using a Creator Function (Zustand style)

The creator function receives `setState`, `getState`, and `api` as arguments, allowing you to define actions directly inside the store.

<CodeBlock language="typescript">
{`import { createStore } from 'soulstate';

export const useBearStore = createStore((setState, getState) => ({
  bears: 0,
  increase: () => setState((state) => ({ bears: state.bears + 1 })),
  reset: () => setState({ bears: 0 })
}));
`}
</CodeBlock>

## `StoreApi` Interface

`createStore` returns a `StoreApi` object with the following methods:

<CodeBlock language="typescript">
{`export interface StoreApi<T extends State> {
  getState: () => T;
  setState: (updater: PartialState<T>, replace?: boolean, sync?: boolean) => void;
  subscribe: <S>(
    selector: Selector<T, S>,
    listener: Listener<S>,
    options?: SubscribeOptions<S>
  ) => () => void;
  computed: {
    <S>(selector: Selector<T, S>, name?: string): Computed<S>;
    <A, B, R>(a: Computed<A> | Selector<T, A>, b: Computed<B> | Selector<T, B>, combiner: (a: A, b: B) => R): Computed<R>;
  };
  beginTransaction: () => void;
  commitTransaction: () => void;
  rollbackTransaction: () => void;
  enableInstrumentation: (options?: RuntimeInstrumentation) => void;
  getMetrics: () => MetricsSnapshot;
  destroy: () => void;
}
`}
</CodeBlock>

## Store Methods

### `store.getState()`

Returns the current state of the store.

- **Signature**: `() => T`
- **Behavior**: Returns the current state. If a transaction is active, returns the buffered state from the transaction engine.

### `store.setState()`

Updates the store's state.

- **Signature**: `(updater, replace?, sync?) => void`
- **Parameters**:
  - `updater`: Partial state object or a function returning partial state.
  - `replace` (optional): If true, replaces the entire state object instead of merging.
  - `sync` (optional): If true, bypasses the microtask scheduler and propagates changes immediately to all subscribers.

### `store.subscribe()`

Subscribes to state changes with granular dependency tracking.

- **Signature**: `<S>(selector, listener, options?) => () => void`
- **Options**:
  - `equalityFn`: Custom comparison function (defaults to `Object.is`).
  - `fireImmediately`: If true, executes the listener immediately with the current state upon subscription.

### `store.computed()`

Creates a derived, memoized state node that is part of the reactive graph.

- **Signature**: `(selector, name?)` or `(...deps, combiner)`
- **Behavior**: Computed nodes are lazily evaluated and cached. They recompute only when their dependencies change, in deterministic topological order.

### `store.beginTransaction()` / `commitTransaction()`

Buffers multiple updates into a single atomic propagation cycle. Ideal for complex multi-step state transitions.

### `store.enableInstrumentation()`

Enables systems-grade diagnostics for profiling the runtime propagation and selector performance.


<Admonition type="info" title="Granular Reactivity">
  <p>SoulState uses a <code>DependencyGraph</code> to track which keys are accessed by your selector. It only executes the listener if one of the accessed keys has changed in the state.</p>
</Admonition>

<Admonition type="success" title="Performance">
  <p>Subscriptions are managed using a doubly linked list for $O(1)$ add/remove operations, ensuring SoulState scales effortlessly to thousands of subscribers.</p>
</Admonition>
